# SeedUp 설문 저장 로직 구조 개선 요청

현재 설문 저장 로직에서 question_id 매핑이 어긋나는 문제가 발생하고 있다.
예를 들어 MAX_HOLDINGS(Q5)의 답변이 저장되지 않거나,
질문 순서 기반으로 잘못된 question_id에 저장되는 문제가 있다.

이 문제를 구조적으로 해결하기 위해,
설문 답변 저장 방식을 다음과 같이 변경해줘.

------------------------------------------------------------
[핵심 원칙]
------------------------------------------------------------

1) 프론트는 설문 질문을 GET /survey/questions 로 받아온다.
2) 설문 저장 시에는 반드시 그 API에서 내려온 question.id 값을 그대로 사용한다.
3) 질문 순서(index)나 하드코딩된 번호로 question_id를 생성하지 않는다.
4) 답변은 반드시 question_id 기준으로 저장한다.

------------------------------------------------------------
[요구 사항]
------------------------------------------------------------

설문 저장 API는 아래 형태의 payload를 받도록 수정한다:

{
  "answers": [
    { "question_id": 1, "value_text": "용돈" },
    { "question_id": 2, "value_text": "3년이상" },
    { "question_id": 3, "value_number": 200000 },
    { "question_id": 4, "value_choice": "LUMP_SUM" },
    { "question_id": 5, "value_number": 200000 },
    { "question_id": 7, "value_number": 10 },
    { "question_id": 8, "value_choice": "MID" },
    { "question_id": 9, "value_text": "일반" }
  ]
}

------------------------------------------------------------
[백엔드 로직 요구사항]
------------------------------------------------------------

1) survey_answers는 (user_id, question_id) 기준으로 upsert 처리한다.
2) question_id가 실제 survey_questions에 존재하는지 검증한다.
3) SINGLE_CHOICE 타입인 경우:
   - survey_questions.options_json에 정의된 값만 허용한다.
4) TEXT / NUMBER / SINGLE_CHOICE 타입에 맞는 value_* 필드만 저장한다.
5) 잘못된 question_id 또는 타입 불일치 시 400 에러 반환한다.

------------------------------------------------------------
[추가 개선]
------------------------------------------------------------

- 프론트에서는 설문 렌더링 시 question.id를 state에 저장한다.
- 설문 제출 시 그 id를 그대로 payload에 포함한다.
- 질문 순서를 기반으로 question_id를 계산하지 않는다.

------------------------------------------------------------
[목표]
------------------------------------------------------------

- 설문 답변 누락 방지
- question_id 매핑 오류 제거
- 조건부 문항(4-1, 4-2) 정확한 저장 보장
- 구조적으로 안정적인 설문 저장 로직 구현
